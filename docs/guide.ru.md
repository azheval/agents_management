# Agent Management

## База данных

Для запуска контейнера с базой данных использовать команду

```make
make db-up
```

или

```cmd
docker-compose up -d
```

При первом запуске необходимо создать таблицы базы данных выполнив миграцию

```make
make db-migrate-fresh
```

или

```cmd
docker exec -i agent_db psql -U admin -d agent_management < db/migrations/001_full_schema.sql
```

## Старт сервера и агента

Для старта сервера использовать команду

```cmd
<путь>/server.exe -config <путь>/config.json
```

Пример файла `config.json` в `server/config.json`.

Для старта агента использовать команду

```cmd
<путь>/agent.exe
```

При старте агента создается служебный файл `agent.id` с уникальным идентификатором агента.

### Права доступа

Для входа в веб-интерфейс используется аутентификация по имени пользователя и паролю. Учетные записи пользователей, роли и их назначения хранятся в базе данных в таблицах `users`, `roles` и `user_roles`.

Настройка прав выполняется в разделе `Access`.

![img_013.png](/docs/img/img_013.png)

В системе используются четыре группы прав:

- `admin` и `full_access` дают полный доступ ко всем разделам и ко всем агентам.
- Глобальные роли `action.*` определяют, какие типы задач пользователь может создавать.
- Права вида `agent:<uuid>:...` определяют, какие действия разрешены для конкретного агента.
- Права `exec_policy:<uuid>:use|view|manage` определяют доступ к сохраненным политикам `EXEC_COMMAND`.

Для создания задачи пользователю требуется одновременно:

- глобальная роль действия, соответствующая типу задачи, например `action.exec_command`;
- право `agent:<uuid>:task_create` для выбранного агента.
- `exec_policy:<policy_uuid>:use`

Для просмотра итоговых `command` и `args` у policy-backed задачи дополнительно требуется:

- `exec_policy:<policy_uuid>:view`

Матрица прав в разделе `Access` позволяет назначать разрешения на пересечении пользователя, агента и операции. Через нее можно отдельно выдавать права на:

- просмотр агента;
- просмотр метрик;
- просмотр задач;
- создание задач;
- изменение расписания задач;
- просмотр уведомлений;
- переключение состояния агента;
- удаление агента.

## Политики EXEC_COMMAND

Сохраненные политики `EXEC_COMMAND` предназначены для безопасного и повторно используемого запуска команд.

Политика хранит:

- имя и описание
- шаблон команды
- шаблон аргументов
- схему параметров

Binding хранит значения для конкретного агента:

- целевой агент
- необязательный override шаблона команды
- необязательный override шаблона аргументов
- JSON со значениями параметров

В шаблонах используется синтаксис двойных фигурных скобок:

- `{{target}}`
- `{{count}}`

Пример политики для `ping`:

- `Command Template`: `ping`
- `Args Template`: `-n,{{count}},{{target}}`

Пример схемы параметров:

```json
[
  {
    "name": "target",
    "label": "Target",
    "type": "text",
    "required": true,
    "editable": false
  },
  {
    "name": "count",
    "label": "Count",
    "type": "number",
    "required": true,
    "editable": true,
    "default_value": "4"
  }
]
```

Пример binding для агента:

```json
{
  "target": "127.0.0.1"
}
```

После резолва задача выполнит:

```text
ping -n 4 127.0.0.1
```

![img_015.png](/docs/img/img_015.png)

Рекомендуемый порядок настройки:

1. Создать пользователя в разделе `Access`.
2. Если пользователю нужен неограниченный доступ, назначить `admin` или `full_access`.
3. Если нужен ограниченный доступ, выдать только необходимые глобальные роли `action.*`.
4. В матрице отметить права только для тех агентов и операций, которые действительно нужны пользователю.
5. Проверить результат, выполнив вход под новой учетной записью.

Роль `agent:<uuid>` сохраняется для совместимости со старыми настройками и означает широкий доступ к агенту. Для новых настроек рекомендуется использовать матрицу точечных прав.

### Создание задачи

Новая задача создается из формы `Tasks` командой `Create new task`.
В форме создания задачи выбираются агенты, которые будут выполнять задачу. Для каждого выбранного агента будет создана своя уникальная задача.
Указывается тип задачи. Для каждого типа задачи требуется заполнить уникальные поля.

#### EXEC_COMMAND

![img_002.png](/docs/img/img_002.png)

Заполняется описание задачи, команда (например, `C:/ws/tech_log/go/techlog-stat/.dist/techlog-stat.exe`), аргументы команды указываются через запятую (например, `sdbl-context,--input,C:/ws/tech_log/logs,--glob,*/*.*,--output,C:/ws/tech_log/out/sdbl_agent,--top,10`).

При использовании шаблонов достаточно выбрать шаблон.

![img_016.png](/docs/img/img_016.png)

#### EXEC_PYTHON_SCRIPT

![img_003.png](/docs/img/img_003.png)

Заполняется описание задачи, выбираются необходимые файлы (не только python файлы), указывается файл оркестратор (Entrypoint) который будет вызван агентом на выполнение.
Все указанные файлы при создании задачи будут помещены в архив, переданы агенту, разархивированы и выполнены.
**Важно**. Python должен быть предварительно установлен на удаленном рабочем месте.

#### FETCH_FILE

![img_004.png](/docs/img/img_004.png)

Заполняется описание задачи, путь к файлу для агента, путь по которому сервер сохранит файл.

#### PUSH_FILE

![img_005.png](/docs/img/img_005.png)

Заполняется описание задачи, путь сохранения файла агентом, файлы передаваемые сервером.

#### AGENT_UPDATE

![img_006.png](/docs/img/img_006.png)

Заполняется описание задачи, указывается файл новой версии агента. Контрольная сумма рассчитывается автоматически.

У любой задачи можно настроить блок `Alert Pipeline` который отправляет поток `stdout` в телеграм.

![img_010.png](/docs/img/img_010.png)

![img_011.png](/docs/img/img_011.png)

Выбирается требуемое расписание выполнения команды, указывается таймаут на выполнение команды:

- Immediate. Задача выполняется немедленно.
- Scheduled. Задача будет выполнена в указанные день и время.
- Recurring. Задача будет выполняться периодически по заданному расписанию.
- Chained. Задача будет выполнена после выполнения предыдущей (выбранной) задачи.
